# 개발사 전달사항

고대표님의 의견으로 한 화면(예: 회원목록)을 개발할 때 `해당 도메인.md` 한 파일과 `운영정책.md` 한 파일, 총 2개 파일만 열면 구현이 가능하도록 `docs4` 문서 구조를 재정리했습니다.

중복 사례는 대부분 제거했으나, V1/V2를 나누는 과정에서 향후 V2 개발 시 V1 기준과 함께 봐야 하는 항목은 의도적으로 양쪽에 중복 기재했습니다. V1 개발 중 V2 문서를 별도로 열어야만 구현 기준을 알 수 있는 상태를 피하기 위한 조치입니다.

ID만 적지 않고, 개발자가 바로 찾아갈 수 있도록 `SCR-M001 / docs4/V1/D02-회원관리/회원관리.md:20`처럼 관련 파일 경로와 라인을 함께 표기했습니다. 라인은 문서 보강 과정에서 변동될 수 있으므로, 최종 납품 직전 한 번 더 갱신합니다.

## 문서 정리 기준

- `docs4/V1`: V1 개발자가 구현 기준으로 보는 문서입니다.
- `docs4/V2`: V2 또는 후속 범위 문서입니다.
- `docs4/_공통`: V1과 V2가 함께 참조하는 공통 권한, 알림, KPI, NFR, 외부연동, 자동화 기준 문서입니다. `_공통` 안에는 V1/V2 항목이 모두 있을 수 있으므로, V1 개발 시에는 먼저 `docs4/V1`의 해당 도메인 문서를 기준으로 확인하고, `_공통`은 그 도메인 문서에서 참조한 공통 기준만 함께 확인합니다.
- `docs4/_V1_정책확인필요.md`: V1 화면 안에 남아 있지만 고객사/판도 확인 후 확정해야 하는 보류 항목입니다.
- `docs4/_정책미확정_확인필요.md`: V1/V2 전체 관점의 미확정 정책 목록입니다.
- `docs4/_scope`: V1/V2 분리 검증 및 내부 확인용 자료입니다. 개발 구현 본문으로 보지 않습니다.

### `_공통`에서 V2가 함께 보이는 이유

`_공통` 문서는 특정 버전 전용 화면 문서가 아니라, 권한 체계, 알림 채널, 자동화, 외부연동, KPI처럼 V1과 V2가 공유하는 기준을 모아 둔 문서입니다. 따라서 V1 개발자가 `_공통`을 열었을 때 V2 경로 또는 V2 항목이 함께 보일 수 있습니다.

이 경우 V2 항목은 V1 구현 지시가 아닙니다. V1 개발 기준은 `docs4/V1` 내부의 도메인 문서가 우선이며, `_공통`은 V1 문서에서 직접 참조한 항목만 V1 구현 기준으로 봅니다. 동일 ID가 V1/V2 양쪽에 모두 존재하면 V1 문서의 경로와 정의를 우선합니다.

## 제거/분리한 내용

- 본문 안의 `docs4 Scope`, `분류 근거`, `주의` 같은 내부 생성 메타는 납품문서 본문에서 제거했습니다.
- 표 잔해처럼 컬럼 의미를 개발자가 추론해야 하는 줄은 문장 또는 정상 표로 재정리했습니다.
- V1 구현 항목이 V2 문서만 참조하도록 남아 있던 부분은 V1 문서 기준 참조로 보정했습니다.
- A/B 테스트는 V1 제외 대상으로 분리했으며, V2 문서에서 후속 범위로 관리합니다.

## 보류 항목 처리

고객사 또는 판도 측 확정이 필요한 정책은 본문에서 임의로 확정하지 않고 `docs4/_V1_정책확인필요.md`에 별도 정리했습니다. 해당 항목은 이번 V1 개발 과정에서 고객사/판도 정책 확인을 받은 뒤 개발 기준으로 반영합니다. 확인 전에는 최종 완료 기준으로 보지 않으며, 확인 결과가 오면 해당 도메인 문서와 운영정책 문서에 확정값을 직접 반영합니다.

외부 Google Spreadsheet 기반 기능은 `판도 스프레드시트 산식 참조`가 필요한 항목입니다. `[매출종합]`, `[월매출]`, `[OT관리-강습/골프]`, `[OT관리-강습]`, `[강습업무보고]`, `[주간월간보고]` 같은 시트명은 현재 운영 원천을 뜻하며, 자동 집계 개발 전 판도에서 컬럼 정의, 산식, 집계 기준을 제공해야 합니다.
