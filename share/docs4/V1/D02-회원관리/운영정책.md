# D02-회원관리 회원관리 — 운영정책

> 개발사용 운영정책입니다. 이 파일만으로 해당 도메인의 운영 기준, 정책값, 예외 기준을 확인할 수 있도록 재구성했습니다.




## 1. 핵심 원칙
- 리드 유입부터 회원 등록, 회원 상세 운영까지의 기준을 하나로 묶는다.
- FC, 매니저, 지점 운영자가 가장 자주 쓰는 흐름을 우선 정의한다.
- 리드 관리
- 회원 목록
- 회원 등록/수정
- 회원 상세
- 회원 이관
- 상담/메모
- 외부 신규 DB 엑셀은 `docs/03_템플릿/지점파일템플릿/2025년 12월 광화문점 신규통합 DB.xlsx`의 `신규통합DB` 시트 형식을 기준으로 한다. 이 파일은 지점/월별 신규 리드 임포트 템플릿 예시이며, 실제 운영 임포트 파일도 같은 컬럼 구조를 사용해야 한다.
- 외부 신규 DB 컬럼 매핑: `경로` 필드는 문의 유형(방문(WI) / 전화문의(TI) / 체험 / 일일입장)으로 매핑하고, `상세 경로1`과 `상세 경로2`는 `가입경로` 단일 필드로 통합 매핑한다. 가입경로 9종은 간판 / 블로그 / 인스타 / 당근 / 지역카페 / 현수막 / 전단지 / 회원소개 / 기타로 고정한다.
- `상세 경로1`과 `상세 경로2` 값이 동시에 들어온 경우 9종 가입경로와 직접 매칭되는 더 구체적인 값을 우선 사용한다. 둘 다 매칭되지 않으면 `기타`로 집계한다. CRM 화면에는 원천 상세경로를 별도 필드로 나누어 노출하지 않고 `가입경로` 하나만 노출한다.
- 외부 신규 DB의 `1차 상담 방식`, `2차 상담 방식`, `3차 상담 방식` 값은 대면 / 유선 / 부재 3종만 허용한다.
- 리드 단계의 가입경로와 실제 첫 정상 결제 완료 회원의 가입경로는 다를 수 있으므로, 첫 정상 결제 완료 기준 가입경로 분포를 별도 표시한다. 가입경로 집계는 간판 / 블로그 / 인스타 / 당근 / 지역카페 / 현수막 / 전단지 / 회원소개 / 기타 9종을 기준으로 한다.
- 상담 성과 담당자는 리드의 `상담 담당자(FC)`로 관리하고, 매출 성과 담당자는 첫 정상 결제 완료 시점의 `판매 담당자/인센티브 귀속자`로 관리한다. 가입경로는 유입 채널이며 직원을 구분하는 용도로 사용하지 않는다.
- 리드는 전화번호를 기준으로 중복 여부를 먼저 확인한다.
- 이미 회원으로 존재하는 전화번호는 신규 리드 생성보다 "기존 회원 연결"을 우선 제안한다.
- 리드 담당자는 FC를 기본으로 하되, 지점 매니저가 재배정할 수 있다.
- `등록완료`는 첫 정상 결제 완료 시점에만 전환되며, 이때 회원 생성 또는 기존 회원 연결이 반드시 발생해야 한다. 칸반에서 `등록완료` 컬럼으로 이동하는 동작은 상태 저장이 아니라 회원 등록/결제 흐름 시작 트리거다.
- `미전환`은 종료 상태이며, 동일 연락처 재유입 시 기존 리드 재활성화를 기본값으로 한다. 새 리드는 연락처가 다르거나 권한자가 중복 강행 사유를 입력한 경우에만 생성한다.
- TI 문의는 전화 응대에서 끝났는지, 실제 대면 상담으로 이어졌는지를 분리해서 본다.
- WI 문의는 현장 방문 고객의 즉시 등록 전환 성과로 본다.
- `미등록`은 온보딩 미완료 상태이며 `비활성`과 합치지 않는다.




## 2. 상태 / 운영 정의

| 지표 | 산정 기준 | 클릭/필터 동작 |
|---|---|---|
| 활성 회원 수 | `status=active` 회원 수 | 활성 필터 자동 적용 |
| 정책상 조치 대상 회원 수 | HQ-09 / `docs4/V1/D10-본사관리/본사관리.md:1393` 회원 이용권 만료 본사 step 또는 지점 추가 step 중 ON 상태인 step 대상 | HQ-09 만료 알림 대상 필터 적용 |
| 이번 달 만료 수 | 당월 실제 만료일 도래 회원 수 | 당월 만료일 필터 적용 |

### 메인 탭 + 상태 필터

- **상태 필터 탭** (회원 전체 탭 내부): 전체 / 활성 / 만료 / 예정 / 임박 / 홀딩 / 미등록 / 탈퇴 총 8개 탭 + 건수 배지. `전체`는 집계 탭이고, 나머지 7개는 실제 회원 상태값이다. `예정`, `임박`, `탈퇴`도 실제 회원 상태로 관리하며 자동 라벨이나 단순 표시 배지와 혼동하지 않는다. `정지`는 회원 상태값이 아니라 현재 상태 위에 적용되는 이용 제한 플래그로 관리한다.

- **FC가 회원 이용권 만료 정책 step 대상 회원에게 리텐션 상담 시**: 만료 배너 + 결제이력 탭 + 상담이력 탭 + 메시지 액션을 한 화면에서 수행

- **매니저가 회원 클레임 응대 시**: 결제내역(미수·환불·링크 상태) + 출석이력 + 상담메모를 즉시 조회해 사실관계 파악

- 회원 기본 정보 영역에는 이름, 등급 배지, 회원 상태를 표시한다. 회원 상태값은 활성 / 만료 / 예정 / 임박 / 홀딩 / 미등록 / 탈퇴를 사용한다.

- 만료일까지 남은 일수는 D-day 형식으로 표시한다.

- [초기화]: 입력한 내용이 있을 때만 활성, 확인 후 전체 초기화 (DLG-M008 / `docs4/V1/D02-회원관리/회원관리.md:1603`)

- 회원 등록 단계에서는 Step 1 필수값이 완료되어야 Step 2 입력 영역이 활성화된다.

- 3. 중복 없음 → 초록 체크 표시 → [다음] 활성화

- 2. 기존 정보 자동 채움 + Step 1/2 모두 진입 가능 (단계 잠금 없음)

- **연락처 중복 시 (DLG-M006 / `docs4/V1/D02-회원관리/회원관리.md:1465`)**: 동일 연락처가 활성 회원에 존재 → 모달에 기존 회원 카드(이름·이용권 상태) 표시 + [기존 회원으로 이동] / [강제 등록(법인 동일 번호 등)] / [닫기] 3선택지

- **회원이 잔여 횟수 다 쓰고 이관 결정**: 소진 후 이관 — 만료일까지 현재 지점 사용, 만료 후 새 지점에서 신규 시작

- **본사 가맹점 정책 변경에 따른 일괄 이관**: 폐점·합병 시 슈퍼관리자가 다수 회원 이관 (CSV 일괄 미지원, 1명씩 진행)

- 이관 대상 회원의 이름, 연락처, 현재 지점, 상태, 이용권 종류, 만료일.

- 회원 이관 대상 목록에는 이름, 연락처, 현재 지점, 상태, 이용권 종류, 만료일을 표시한다.

- 본사 슈퍼관리자는 가맹점 폐점에 따른 다수 회원 이관을 처리할 수 있다. V1에서는 CSV 일괄 이관을 지원하지 않고 1명씩 순차 처리하며, 회원 동의를 받은 인원만 이관한다. 미동의자는 환불 후 탈퇴 처리한다.




## 3. 자동 7종 세그먼트 운영 기준

| 라벨 | 판정 기준 | 운영 기준 |
|---|---|---|
| 신규 | 첫 정상 결제 완료일로부터 30일 이내 | 회원 등록은 결제 없이 완료되지 않으므로 등록일이 아니라 첫 결제 완료일을 기준으로 한다. |
| 만료후미등록 | 마지막 회원 이용권 만료일 +60일 경과, 만료 이후 재등록 결제 없음 | 재등록 결제 완료 즉시 후보에서 제외한다. |
| 이탈위험 | 활성 회원이 최근 30일 이상 방문/출석 없음 | 키오스크 입장, 수업 출석, PT/GX 참여, 수동 출석을 모두 포함하고 2시간 이내 중복 입장은 1회 처리한다. |
| 만료임박 | HQ-09 / `docs4/V1/D10-본사관리/본사관리.md:1393` 회원 이용권 만료 본사 step 또는 지점 추가 step 중 ON 상태인 step 대상 | 결제기한 만료·락커 만료 정책과 분리한다. |
| 관심필요 | 최근 90일 이내 종합평가 기록이 없거나 상담 기록이 없는 회원 | 지속적인 관심이 필요한 회원을 넓게 포착하기 위해 둘 중 하나만 없어도 후보에 포함한다. |
| 충성 | 누적 결제 기간 12개월 이상 + 골드 이상 등급 | 홀딩 기간은 제외하고, 지점 이관 시 이전 지점 이용기간을 포함한다. 다른 자동 라벨과 중복 표시 가능하다. |
| 활발 | 최근 30일 이내 방문/출석 8회 이상 | 키오스크 입장, 수업 출석, PT/GX 참여, 수동 출석을 모두 포함하고 2시간 이내 중복 입장은 1회 처리한다. |

- 기본 자동 라벨 우선순위는 `이탈위험 > 만료임박 > 신규 > 활발 > 관심필요 > 만료후미등록`이다.
- `충성`은 보조 라벨로 별도 판정하며 다른 기본 자동 라벨과 동시에 표시할 수 있다.
- 세그먼트 관리, 회원 목록 배지, 회원 상세 헤더, 본사 통합 KPI는 동일한 판정 기준을 사용한다.




## 4. 권한·책임 분리
- 본사 권한자는 통합 모드에서 전 지점 조회·정책 관리를 수행한다.
- 지점 권한자는 소속 지점 데이터의 일상 운영 처리를 수행한다.
- 변경·삭제·복구·정산·권한 변경은 화면별 최소 권한을 따른다.
- 공통 권한 기준은 `_공통/권한매트릭스.md`를 우선 참조하고, 화면별 차이는 도메인 문서에 따로 표기한다.

## 7. 정책 결정 근거
- 화면별 구현 판단은 `UI 구성`, `액션 / 흐름`, `권한`, `필수 예외처리` 순서로 확인한다.
- 여러 항목 값이 충돌하면 docs2 문서 안에서 하나의 기준값으로 보정해 반영한다.
- 클라이언트 운영 정책값이 확정되지 않은 항목은 `docs4/_정책미확정_확인필요.md`에 별도 관리한다.
