# 우선수정 묶음

`정합성_이슈_리포트.md`의 자동 검출 결과 중 개발사 질의로 이어질 가능성이 높은 항목을 기능 묶음 단위로 정리한다.

## 1. D04 일정 요청 처리 / 알림 묶음

- 관련 기능: `CLS-09`, `CLS-09-06`, `SCR-C009`, `SCR-090`, `NFR-19`
- 문제: `탭 배지 건수`, `대시보드 알림`의 표시 위치와 카운트 기준이 짧은 기능명 수준으로 남아 있다.
- 개발사 질문 가능성:
  - 어느 탭의 배지인가?
  - 미처리 건수 기준은 무엇인가?
  - 어느 대시보드에 표시되는가?
  - 클릭하면 어느 화면과 필터로 이동하는가?
- 보정 방향:
  - `CLS-09-06-01`: `SCR-C009 일정 요청 처리 > 대기 탭`에 표시되는 `처리 상태=대기` 건수로 명시한다.
  - `CLS-09-06-02`: `SCR-090 지점 대시보드 > 오늘 처리할 업무/액션 큐` 및 `NFR-19 알림 센터`의 `미처리 일정 요청 N건` 알림으로 명시한다.

## 2. D07 직원관리 / 매니저 인앱 알림 묶음

- 관련 기능: `STF-02`, `PAY-STF-03`, `NFR-19`
- 보정 상태: `docs/admin`과 `docs2`에서 `매니저 인앱 알림`으로 정의 완료.
- 정본 의미: `manager` 권한 계정으로 로그인한 사용자에게 관리자 웹 내부 화면에서 알림 배너, 토스트, 알림 패널 형태로 표시되는 시스템 알림이다.
- 화면 기준: 특정 대시보드 화면(SCR-090/SCR-098)에 고정 노출되는 의미가 아니다.
- 클릭 이동:
  - `STF-02`: 직원 상세의 인사 메모, OT 진행 현황, NPS 카드 또는 SCR-C013 후기로 이동.
  - `PAY-STF-03`: 급여 명세서 화면의 해당 직원·월·미열람 또는 발송 실패 필터로 이동.

## 3. D09 설정관리 / IoT 오프라인 임계값 묶음

- 관련 기능: `IoT-02`, `SCR-083`, `docs2/D09 설정관리`
- 문제: `docs/admin`에는 5분 헬스체크/오프라인 알림 기준이 남아 있고, `docs2`에는 기본 30분 + 지점 설정 N분으로 정리되어 있다.
- 개발사 질문 가능성:
  - 5분과 30분 중 어떤 값이 정본인가?
  - 헬스체크 주기와 오프라인 알림 임계값이 같은 값인가?
  - 지점 설정값으로 바꿀 수 있는 것은 어느 값인가?
- 보정 방향:
  - 헬스체크 주기와 오프라인 알림 임계값을 분리한다.
  - 헬스체크 주기는 시스템 내부 주기, 오프라인 알림 임계값은 지점 설정값으로 정의한다.
  - 최종 기본값은 고객사/판도 정책 확인 후 `docs/admin`과 `docs2`에 동일 반영한다.

## 4. Route 책임 중복 묶음

- 관련 route:
  - `/`
  - `/body-composition`
  - `/locker/management`
  - `/rooms`
  - `/message/auto-alarm`
  - `/settings/iot`
- 문제: 하나의 route에 서로 다른 화면 책임이 남아 있다. 일부는 역할별 분기 또는 탭일 수 있으나, 문서상으로 의도된 하위 모드인지 충돌인지 명시가 필요하다.
- 보정 방향:
  - route 자체는 개발사가 정해도 되지만, 기획 문서에는 화면 책임을 분리해 적는다.
  - 같은 route를 유지할 경우 `탭/하위 모드/권한별 분기`라고 명시한다.
  - 다른 화면이면 route 또는 화면 ID 설명을 분리한다.

## 5. ID 참조 누락 묶음

- 주요 후보:
  - `SCR-G003`
  - `SCR-PAY`
  - `SCR-M013`
- 문제: 문서에서 참조되지만 `docs/admin` 관리자 화면 정의에서 확인되지 않는다.
- 보정 방향:
  - 실제 화면이면 화면설계서 ID를 생성한다.
  - 오기이면 현재 정본 화면 ID로 교체한다.
  - 외부/회원앱 화면이면 관리자 문서 참조가 아님을 명시하고 관리자 동작 기준만 남긴다.

## 6. 권한 축약 표현 묶음

- 문제: `owner+`, `manager+`, `센터장+`, `매니저+` 같은 축약 표현이 남아 있다.
- 개발사 질문 가능성:
  - 정확히 어떤 역할까지 포함되는가?
  - 버튼 노출과 실행 권한이 같은가?
- 보정 방향:
  - 개발용 문서에서는 축약 표현 대신 포함 역할을 풀어 쓴다.
  - 예: `owner 이상` → `owner / primary / superAdmin`
  - 예: `manager 이상` → `manager / owner / primary / superAdmin`
