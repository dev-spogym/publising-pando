# docs2 전체 정합성/Drift 재점검

- 기준일: 2026-05-23
- 범위: `docs2/**/*.md` 전체. 단, `reports/**`, `registry/**`, `scripts/**`, `정합성체크/**` 기존 보고서는 원문 기획 drift 판단에서 제외
- 기준: 개발팀이 문서만 보고 화면 동작, 버튼 결과, 상태값, 권한, 자동화 실행 주체/시점, 고객사 확인 대기 여부를 오해 없이 구현할 수 있는지 확인
- 원문 수정 여부: 최초 점검 시에는 원문을 수정하지 않고 결과만 기록했다. 이후 사용자 결정에 따라 아래 `0. 사용자 결정 반영 결과` 항목을 원문 기획 문서에 반영했다.

## 1. 자동 검사 결과

`python scripts\docs_fact_check.py` 실행 결과:

| 구분 | 건수 | 판정 |
|---|---:|---|
| ERROR | 0 | `registry/facts.yml` 기준으로 자동 탐지 가능한 drift는 없음 |
| WARN | 496 | 대부분 `facts.yml`에 아직 등록되지 않은 자동화/배치 문장 |

WARN 분포:

| category | 건수 | 해석 |
|---|---:|---|
| `new_automation` | 493 | 원문에는 자동화/배치가 있으나 registry 정본에 아직 충분히 승격되지 않음 |
| `new_timing_condition` | 1 | 신규 시점/조건 후보 |
| `new_external_integration` | 1 | 신규 외부연동 후보 |
| `new_exception` | 1 | 신규 예외처리 후보 |

자동 검사상 ERROR는 0이지만, WARN 496은 “drift 없음”이 아니라 “정본 registry가 아직 자동화 영역을 충분히 포괄하지 못한다”는 의미다. 개발 공유 전에는 안정된 자동화/배치명과 실행 조건을 `registry/facts.yml`에 추가하거나, 의도적으로 registry 관리 대상에서 제외하는 기준이 필요하다.

## 2. 실제 수정 필요 Drift 후보

### 2.1 HQ-09 회원 이용권 만료 step의 대상 산출 시점/소유자가 문서마다 다르게 읽힘

| 문서 | 현재 문장 |
|---|---|
| `D02-회원관리/회원관리.md:101` | `NFR-05 회원 이용권 만료 알림 배치`가 매일 새벽 03:00에 카드 카운트와 만료 배너 결과를 갱신 |
| `D08-마케팅/마케팅.md:271`, `:356`, `:1414`, `:1504` | `회원 이용권 만료 step 일배치`가 매일 04:00에 대상 추출 + 설정 시각 발송 |
| `D10-본사관리/본사관리.md:119` | `만료 감지 배치`가 매일 00:00에 HQ-09 회원 이용권 만료 step 대상과 만료 회원 분류 |
| `D04-수업관리/수업관리.md:565` 등 | NFR-05 결과를 매일 04:00 이후 배지와 통계에 반영 |

문제:

- 같은 `HQ-09 회원 이용권 만료 step` 대상을 D02, D08, D10이 각각 다른 시각에 직접 추출/감지하는 것처럼 보인다.
- 개발팀이 각 도메인별 독립 배치를 만들면 회원 목록 배너, 자동 알림 대상, 대시보드 위젯의 대상 수가 하루 동안 서로 달라질 수 있다.
- D04는 “04:00 이후 NFR-05 결과 소비”로 정리되어 있어 비교적 명확하지만, D08/D10은 “소비”가 아니라 “추출/감지”로 쓰여 중복 산출 위험이 있다.

권장 해결:

- 정본을 하나로 고정한다.
  - 예: `D02/NFR-05`가 매일 03:00에 HQ-09 회원 이용권 만료 step 결과를 산출한다.
  - `D08`은 04:00에 새로 대상 추출을 하지 않고, D02/NFR-05 산출 결과를 읽어 발송 큐를 생성한다고 쓴다.
  - `D10`은 00:00 감지가 아니라 대시보드 캐시/조회 갱신으로 표현하거나, D02 결과를 소비한다고 쓴다.
- 사용자 확인 필요: 최종 정본 시각을 `03:00`, `04:00`, `00:00` 중 무엇으로 둘지 결정해야 한다. 현재 문서 흐름상 `D02 03:00 산출` + `D08/D04 04:00 이후 소비`가 가장 안전하다.

### 2.2 D04 수업 출석에서 “잔여 0/만료 차단”과 “만료 회원 출석 처리”가 충돌함

| 문서 | 현재 문장 |
|---|---|
| `D04-수업관리/운영정책.md:24` | 예약 시 잔여 횟수 보유 여부만 검증하며, 잔여 0회이면 예약·출석을 차단 |
| `D04-수업관리/수업관리.md:878` | 이용권 만료 회원 출석 시 “이용권 만료, 차감 실패” 후 출석 처리 + 횟수 조정 안내 |
| `D04-수업관리/수업관리.md:1294` | PT 차감 실패(이용권 만료) 시 출석은 처리, 차감 실패 알림 |

문제:

- 한쪽은 잔여 0회이면 예약·출석을 차단한다고 하고, 다른 쪽은 이용권 만료 상태에서도 출석은 처리한다고 한다.
- “만료됐지만 예약은 이미 있었던 회원”, “잔여 0이 된 회원”, “관리자가 강제로 출석 처리해야 하는 예외”를 구분하지 않으면 수업권 없이 출석 완료가 생길 수 있다.

권장 해결:

- 기본 정책은 `잔여 0 또는 이용권 만료 상태에서는 출석 처리 차단`으로 두는 것이 현재 운영정책과 맞다.
- 운영 보정이 필요하면 별도 예외로 분리한다.
  - 예: `권한자 수동 보정 출석` 또는 `차감 실패 보정` 케이스에서만 사유 입력 후 출석 기록 가능.
  - 이 경우에도 “다음 결제 후 차감 대기”는 만들지 않는다고 유지한다.
- 사용자 확인 필요: 만료/잔여 0 상태에서 현장 사유 입력 후 강제 출석 처리를 허용할지 여부.

### 2.5 D08 캠페인과 A/B 테스트가 같은 자동화명을 쓰지만 종료 상태가 다름

| 문서 | 현재 문장 |
|---|---|
| `D08-마케팅/마케팅.md:659` | 캠페인 상태: 준비·진행·종료 |
| `D08-마케팅/마케팅.md:715` | `종료일 자동 종료 배치`: 진행 → 종료 |
| `D08-마케팅/마케팅.md:924` | A/B 테스트 상태: 준비·진행·완료 |
| `D08-마케팅/마케팅.md:984` | `종료일 자동 종료 배치`: 진행 → 완료 + 통계 검증 |

문제:

- 캠페인과 A/B 테스트가 별도 엔티티라면 상태값이 달라도 된다.
- 하지만 자동화명이 둘 다 `종료일 자동 종료 배치`라서, 같은 배치가 어떤 상태값을 써야 하는지 불명확하다.

권장 해결:

- 자동화명을 분리한다.
  - 캠페인: `캠페인 종료일 자동 종료 배치`, 결과 `진행 → 종료`
  - A/B 테스트: `A/B 테스트 종료일 자동 완료 배치`, 결과 `진행 → 완료 + 통계 검증`
- 사용자 결정 불필요. 현재 상태값 자체는 문맥상 분리되어 있으므로 배치명만 명확히 하면 된다.

문제:

  - 예: `고객사 확인 후 테스트 생성·삭제, 결과 분석, 승리 버전 채택`
- 사용자 결정 불필요. 이미 `_정책미확정_확인필요.md`와 본문이 고객사 확인 후 개발로 정리되어 있다.

### 2.7 D06 운동복 “일일 자동 만료/강제 회수” 정책의 소유 위치와 기본값이 없음

| 문서 | 현재 문장 |
|---|---|
| `D06-시설관리/시설관리.md:551` | 일일 자동 만료 처리(당일 반납 정책): 매일 22:00, 당일 미반납 운동복 알림 + 강제 회수 옵션 |
| `D09-설정관리/설정관리.md:161-167` | 락커·사물함 자동 회수 정책만 정의 |
| `D06-시설관리/운영정책.md:30-32`, `:65-72` | 락커/사물함 자동 회수는 D09 정책 ON일 때만 실행 |

문제:

- 운동복은 락커/사물함과 다른 자산인데, `강제 회수 옵션`의 ON/OFF, 기본값, 제어 권한, 실행 결과가 없다.
- 개발팀은 운동복 미반납을 22:00에 자동으로 회수 처리해야 하는지, 알림만 보내야 하는지, 운영자가 수동 회수해야 하는지 판단할 수 없다.

권장 해결:

- 둘 중 하나로 정본화해야 한다.
  - A안: 운동복 자동 만료/강제 회수도 D09 지점 자동화 정책에 추가한다. 기본값은 OFF, Owner(지점장) 제어.
  - B안: 운동복은 D06 화면 내 로컬 정책으로 관리한다고 명시한다. 기본값/권한/실행 조건을 D06에 적는다.
- 사용자 확인 필요: 운동복 자동 회수 정책을 D09 공통 자동화에 둘지, D06 운동복 화면의 로컬 정책으로 둘지 결정해야 한다.

| FC 리드 전환과 회원 등록 권한 | drift 아님. FC는 회원 등록 화면에 직접 진입하지 않고 등록 요청을 생성하며, Owner(지점장)/매니저가 SCR-M002를 이어받음 |

| `/message/auto-alarm` route 중복 | drift 아님. SCR-072 설정과 SCR-072A 운영현황은 같은 페이지 탭 |

## 4. 개발 공유 전 정리 우선순위

1. `HQ-09 회원 이용권 만료 step`의 산출 주체와 실행 시각을 하나로 고정해야 한다.
2. D04에서 만료/잔여 0 회원의 출석 처리 기본 정책과 강제 보정 예외를 분리해야 한다.
3. D05 GX 세부종목, D06 운동룸 유형처럼 값 목록이 다른 항목은 라벨을 즉시 통일해야 한다.
4. D08 A/B 테스트는 고객사 확인 전 대기 조건이 권한표와 자동화명에도 들어가야 한다.
5. D06 운동복 일일 만료/강제 회수는 정책 소유 위치와 기본값이 필요하다.
6. `registry/facts.yml`은 현재 상태값/수량 중심은 유효하지만 자동화/배치 정본이 부족하다. WARN 496을 줄이려면 자동화 registry를 보강해야 한다.
